# EnvOps 数据库资源连通性检测设计

## 1. 背景与目标

当前 `v0.0.4` 数据库资源页只支持数据库资产目录维护，`connectivityStatus` 仍是人工维护字段。用户希望把它升级为可执行的真实连通性检测能力，但仍保持数据库资源模块作为资产目录页面，而不是扩展成数据库治理平台。

本次目标是为数据库资源管理补齐真实数据库连接检测能力，支持页面发起单个检测和批量检测，并将检测结果回写到当前数据库资产记录。

## 2. 本次设计结论

### 2.1 用户确认的功能边界

- 采用最小可用方案，不做任务中心化异步检测
- 只有真正完成数据库连接并通过认证，才算检测成功
- 不新建数据库专用凭据体系，连接凭据直接维护在数据库资产上
- 六种数据库类型全部支持真实连接检测：`mysql`、`postgresql`、`oracle`、`sqlserver`、`mongodb`、`redis`
- 检测成功回写 `online`，检测失败回写 `offline`
- 批量检测支持三种入口：已选记录、当前页、全部筛选结果
- 缺少连接凭据的记录在批量检测中跳过并记为失败，返回明确原因
- 编辑数据库资产时不回显旧密码，密码留空表示沿用旧值，填写才覆盖
- 不落库保存失败原因，只在本次检测结果中返回明细
- 当前只支持基础直连，不支持 SSL、连接串、副本集、Sentinel、Cluster 等高级连接能力
- 批量检测结果同步返回汇总和逐条明细

### 2.2 交付目标

交付完成后，数据库资源页应具备以下能力：

1. 用户可以对单条数据库资产发起真实连接检测
2. 用户可以对已选记录、当前页、全部筛选结果发起批量检测
3. 后端根据数据库类型执行真实连接和最小确认动作
4. 检测结果会自动回写 `connectivityStatus` 与 `lastCheckedAt`
5. 页面可以展示单检结果与批量结果明细
6. 文档口径从“完全手工维护连通性状态”更新为“支持真实检测，但边界明确”

## 3. 总体方案

本次功能继续落在现有资产模块中实现，不拆新子系统。数据库检测本质上是数据库资产上的一个动作，应与当前数据库资源页面、资产接口和资产表结构保持同一条主链路。

整体采用同步请求内执行方案：

- 单个检测由单条接口同步执行并返回结果
- 批量检测由批量接口同步执行并返回汇总与逐条结果
- 后端在当前请求生命周期内依次完成检测、状态回写和结果聚合
- 前端收到结果后刷新当前列表，展示更新后的状态与最近检测时间

本方案避免引入任务中心、轮询、状态机和额外的批处理基础设施，优先保证真实检测能力可以尽快落地并与现有资产目录模式兼容。

## 4. 后端设计

### 4.1 数据模型调整

在现有 `asset_database` 表上新增连接字段：

- `connection_username`
- `connection_password`

设计原则：

- 连接字段直接归属数据库资产记录，不新建数据库专用凭据中心
- `connection_password` 采用现有加密保护能力加密后入库
- 不新增 SSL、连接 URL、高级参数、副本集、Sentinel、Cluster 等字段
- 继续复用现有 `database_name`、`instance_name`、`host_id`、`port`、`database_type`、`connectivity_status`、`last_checked_at`

编辑策略：

- 编辑时不回显历史密码
- 表单密码为空表示沿用旧密码
- 只有用户填写新密码时才覆盖旧值

### 4.2 应用层职责拆分

为避免把检测逻辑继续堆入 `AssetApplicationService`，新增以下职责：

- `DatabaseConnectivityService`
  - 提供单个检测与批量检测入口
  - 负责解析检测目标、逐条执行、汇总结果、回写状态
- `DatabaseConnectionFactory`
  - 根据 `databaseType` 选择具体检测实现
- `DatabaseConnectivityChecker`
  - 定义统一的数据库检测接口
- 各数据库类型检测器
  - `MySqlConnectivityChecker`
  - `PostgreSqlConnectivityChecker`
  - `OracleConnectivityChecker`
  - `SqlServerConnectivityChecker`
  - `MongoDbConnectivityChecker`
  - `RedisConnectivityChecker`

`AssetApplicationService` 继续负责数据库资产 CRUD 与查询，检测动作通过新增服务协作完成，避免单个 service 继续膨胀。

### 4.3 接口设计

新增以下接口：

- `POST /api/assets/databases/{id}/connectivity-check`
  - 对单条数据库资产执行检测
- `POST /api/assets/databases/connectivity-check:selected`
  - 对用户勾选的数据库资产批量检测
- `POST /api/assets/databases/connectivity-check:page`
  - 对当前页记录批量检测
- `POST /api/assets/databases/connectivity-check:query`
  - 对当前筛选结果全部记录批量检测

统一返回结构：

- `summary`
  - `total`
  - `success`
  - `failed`
  - `skipped`
- `results`
  - `databaseId`
  - `databaseName`
  - `databaseType`
  - `environment`
  - `status`，取值如 `success` / `failed` / `skipped`
  - `message`，用于返回失败原因或跳过原因
  - `connectivityStatus`
  - `checkedAt`

其中：

- 单条检测也复用同一结果结构，只是 `results` 仅一条
- `selected` 接口按数据库 ID 列表执行
- `page` 接口由前端提交当前页记录 ID 列表，避免后端重新推断分页上下文
- `query` 接口按当前筛选条件查询所有命中记录后执行，不受当前分页限制

### 4.4 结果回写规则

统一按以下规则处理：

- 成功建立真实连接并完成认证，写 `online`
- 任意失败场景，写 `offline`
- 缺少连接用户名或密码，记为 `skipped`，不做真实连接，返回失败原因
- 所有实际尝试过连接的记录都更新 `lastCheckedAt`
- `warning` 继续保留为人工维护状态，不由检测流程自动写入

本次不落库保存最近失败原因，失败原因只在本次接口响应中返回，供页面结果弹窗展示。

### 4.5 六种数据库的真实检测策略

所有数据库类型都必须发起真实连接，TCP 端口可达但未完成认证不算成功。

- **MySQL**：使用 JDBC 获取连接并执行最小校验
- **PostgreSQL**：使用 JDBC 获取连接并执行最小校验
- **Oracle**：使用 JDBC 获取连接并执行最小校验
- **SQL Server**：使用 JDBC 获取连接并执行最小校验
- **MongoDB**：使用官方客户端建立连接并执行最小命令
- **Redis**：使用客户端完成认证并执行 `PING` 等最小命令

连接参数统一使用基础直连模型：

- 主机地址来自关联主机
- 端口来自数据库资产
- 用户名来自数据库资产连接字段
- 密码来自数据库资产连接字段
- 数据库名或实例名尽量复用当前 `databaseName` / `instanceName`

明确不支持：

- SSL
- 连接串自定义参数
- MongoDB 副本集
- Redis Sentinel / Cluster
- Oracle 高级服务发现
- SQL Server 高级扩展连接参数

### 4.6 批量执行策略

第一版采用串行执行：

- 单次请求内按目标顺序逐条执行
- 每条检测设置合理超时
- 单条失败不影响后续记录执行
- 最终统一汇总返回

选择串行执行的原因：

- 第一版目标是优先保证六种数据库真实连接的正确性
- 并发会显著增加线程池、超时治理和资源释放复杂度
- 本次范围不引入异步任务系统，不适合同时拉高批处理复杂度

### 4.7 安全要求

- `connection_password` 入库前必须加密保存
- 连接密码只在检测执行时解密并短暂驻留内存
- 任意接口响应都不得返回连接密码
- 编辑接口保留“密码留空沿用旧值”的语义
- 日志中不打印密码或完整连接敏感信息

## 5. 前端设计

### 5.1 页面动作入口

继续基于 `frontend/src/views/asset/database/index.vue` 扩展，不新建独立检测页面。

列表页新增以下入口：

- 行级 `检测` 按钮，用于单条检测
- 顶部 `批量检测已选`
- 顶部 `检测当前页`
- 顶部 `检测全部筛选结果`

设计目标：

- 单条排障可以直接在行上完成
- 小批量排障通过已选记录完成
- 当前页检测适合用户只关注眼前这一页数据的场景
- 全部筛选结果检测适合环境级或类型级巡检场景

### 5.2 批量交互规则

- 表格支持多选
- 未选中记录时，`批量检测已选` 按钮置灰
- `检测当前页` 仅针对当前分页可见记录
- `检测全部筛选结果` 直接复用当前筛选条件，不受分页限制
- 所有批量动作执行前都弹确认框，并明确提示本次将检测的记录范围和数量

### 5.3 编辑表单调整

在数据库资产创建/编辑抽屉中补充连接字段：

- 连接用户名
- 连接密码

交互规则：

- 编辑时连接密码输入框默认为空
- 留空表示沿用旧密码
- 用户输入新密码后才覆盖旧值
- 页面不提供历史密码查看能力

### 5.4 结果反馈方式

单个检测：

- 直接使用 message 提示成功或失败
- 完成后刷新当前列表

批量检测：

- 使用结果弹窗展示本次检测结果
- 展示汇总：总数、成功数、失败数、跳过数
- 展示逐条结果列表：数据库名称、类型、环境、执行结果、失败原因
- 用户关闭弹窗后保留当前筛选条件并刷新列表

### 5.5 页面语义更新

页面文案需要从“连通性状态手工维护”调整为更准确的口径：

- 支持真实连接检测
- 检测结果自动回写 `online/offline`
- `warning` 仍为人工标记
- 支持单个检测、批量检测已选、检测当前页、检测全部筛选结果
- 仅支持基础直连，不支持 SSL、连接串、集群、副本集、Sentinel 等高级能力

需要同步更新以下文档，使其与实现保持一致：

- `README.md`
- `docs/envops-项目详细说明.md`
- `docs/envops-开发技术说明.md`
- `docs/envops-用户操作手册.md`
- `release/0.0.4-release-notes.md`

文档必须明确：

- 数据库资源已支持真实连接检测
- 检测入口包括单检、已选批检、当前页批检、全部筛选结果批检
- 支持数据库类型为六种：`mysql`、`postgresql`、`oracle`、`sqlserver`、`mongodb`、`redis`
- 成功回写 `online`，失败回写 `offline`
- `warning` 仍为人工维护状态
- 仅支持基础直连，不支持高级连接能力

## 8. 明确不做的范围

为保证本次仍是最小可交付批次，以下内容不纳入本次实现：

- 数据库专用凭据中心
- 异步任务化检测
- 检测历史审计与检测日志查询页面
- 最近失败原因落库
- SSL 支持
- URL 参数与高级连接串治理
- MongoDB 副本集支持
- Redis Sentinel / Cluster 支持
- 数据库连接池与长期驻留访问层
- 将数据库资源页扩展成数据库治理或 DBA 平台

## 9. 风险与约束

- 六种数据库类型需要引入对应客户端依赖，需控制最小引入范围
- Oracle 与 SQL Server 驱动配置需要注意与当前构建环境兼容
- “全部筛选结果检测”在数据量较大时请求耗时会增长，这是同步方案的已知取舍
- 当前版本不做异步任务化，因此前端需要在交互上清晰提示批量检测范围与等待预期

## 10. 实现原则

- 只补齐真实连接检测必需能力，不扩展成更大平台
- 以资产目录主链路为中心设计，不拆无必要的新系统
- 优先保证真实连接、状态回写和结果可解释性
- 文档口径必须与实际实现完全一致
