# EnvOps v0.0.6 Unified Task Center 设计

## 1. 背景与目标

当前 EnvOps 在三个方向上已经有了真实能力，但它们还分散在各自模块里：

- Deploy 已有真实任务链路
- 数据库资源已支持真实连通性检测
- Traffic 已完成 `REST + weighted_routing + preview/apply/rollback` 的最小真实 MVP

现在最明显的断层变成了 Task Center。

Task Center 当前仍然只是 deploy-only 队列视图。文档和页面都明确承认它不是跨域统一队列。这让用户在真实使用时会遇到一个很别扭的问题：真实任务已经存在，但没有一个统一地方能看到它们。

v0.0.6 的目标不是一次做成完整任务平台，而是把 Task Center 从“Deploy 专属队列”收敛成一个**最小但真实的统一任务中心**，先统一纳入三类真实任务：

1. Deploy 任务
2. 数据库连通性检测任务
3. Traffic 动作任务，也就是 `preview` / `apply` / `rollback`

本次版本只承诺以下结果：

- Task Center 列表统一展示三类任务
- 三类任务使用统一的基础字段和统一状态口径
- Task Center 内可打开轻量详情抽屉查看任务摘要和结果预览
- 用户仍可从 Task Center 深链回原模块查看更深细节
- 成功和失败任务都进入统一任务中心
- 历史只补 Deploy，数据库检测和 Traffic 从 v0.0.6 起开始纳管

## 2. 用户确认的范围边界

### 2.1 本次纳入范围

- Task Center 统一展示 `deploy`、`database_connectivity`、`traffic_action` 三类任务
- 统一任务列表字段：任务类型、任务名称、状态、发起人、开始时间、结束时间、摘要、操作
- Task Center 支持最小筛选：任务类型、状态、时间范围、关键词
- 任务点击后打开轻量详情抽屉，而不是直接跳走
- 详情抽屉展示基础信息、摘要、失败原因或汇总信息、任务特定预览
- 数据库批量检测按“一批一条任务”记录
- 成功与失败都记录
- 历史只补 Deploy
- 数据库检测和 Traffic 动作从 v0.0.6 起记录新增任务
- Deploy 历史任务与后续新任务都进入统一任务中心

### 2.2 明确不做的范围

- 完整统一详情页
- 任务重试
- 任务取消
- 多任务编排
- 全历史补录
- 新执行引擎抽象
- Deploy / 数据库检测 / Traffic 原模块执行模型重写
- Traffic 多插件、多策略矩阵
- 数据库检测子任务列表化展示

## 3. 设计结论

本次采用 **统一任务中心表 + 模块侧写入适配 + 统一列表 + 轻量详情抽屉 + 原模块深链** 的方案。

### 3.1 为什么选这个方案

1. **优于运行时现查现拼**
   - 如果 Task Center 每次查询时临时去 Deploy、数据库检测、Traffic 三边拼装，分页、排序、筛选、失败摘要都会越来越难维护。
   - 单独维护统一任务中心表，查询模型更稳，后续扩展第 4 类任务也更简单。

2. **优于直接做完整统一任务平台**
   - 如果一开始就要统一详情、统一控制、统一重试/取消，就会把权限、幂等、执行器、历史数据清洗一起拉进来。
   - 当前版本最值钱的是先把“统一看见”做实，不是把平台一次做满。

3. **符合当前仓库节奏**
   - 过去几个版本都在把“看起来像功能”的东西收敛成“最小但真实的能力”。
   - Task Center 的最小真实化，和数据库连通性、Traffic MVP 的路线一致。

### 3.2 核心原则

- Task Center 只记录**真实发起过**的任务，不记录不可执行对象
- 模块保留自己的真实执行模型，Task Center 只做统一投影和统一查询
- 任务真正进入执行链路时创建记录，结果明确后更新记录
- 失败任务必须保留，且必须带可读原因
- 数据库批量检测在任务中心里按“一批一条任务”呈现
- 轻量详情抽屉只展示足够理解任务的信息，更深内容回原模块

## 4. 当前实现现状与要收口的问题

### 4.1 当前已有能力

- Deploy 已有真实任务执行与任务数据
- 数据库资源已支持单条检测、批量检测已选、检测当前页、检测全部筛选结果
- Traffic 已有真实外部调用和状态回写
- Task Center 页面、路由和测试已存在，但当前口径仍是 deploy-only

### 4.2 当前问题

- 用户不能在一个地方统一看到数据库检测和 Traffic 动作
- 成功与失败记录分散在模块页面与提示消息里，不利于排障与追踪
- 当前 Task Center 的存在感很弱，更像 Deploy 附属视图，不像系统级任务入口
- Deploy 有历史数据，但数据库检测和 Traffic 还没有统一任务沉淀

## 5. 统一任务模型

v0.0.6 不重写三类模块的真实执行模型，而是在其上增加统一任务中心视图模型。

### 5.1 统一任务中心记录建议字段

- `id`
- `taskType`：`deploy` / `database_connectivity` / `traffic_action`
- `taskName`
- `status`：`pending` / `running` / `success` / `failed`
- `triggeredBy`
- `startedAt`
- `finishedAt`
- `summary`
- `detailPreview`
- `sourceId`
- `sourceRoute`
- `moduleName`
- `errorSummary`
- `createdAt`
- `updatedAt`

其中：

- `summary` 用于列表快速理解任务
- `detailPreview` 用于详情抽屉展示轻量预览
- `sourceId + sourceRoute` 用于深链回原模块
- `errorSummary` 用于失败列表和详情快速展示原因

### 5.2 统一状态映射

统一任务中心先收敛成四档状态：

- `pending`
- `running`
- `success`
- `failed`

映射规则：

- Deploy：现有状态映射到统一四档，原始状态文本保留在摘要或详情预览里
- 数据库检测：真实发起后 `running`，全部完成且无失败时 `success`，有失败或整体执行异常时 `failed`
- Traffic：真实发起后 `running`，外部调用成功且状态回写完成后 `success`，任一真实执行失败则 `failed`

### 5.3 不生成任务的场景

以下情况不应进入统一任务中心：

- Traffic 行本来就不可执行，例如插件不支持、策略不支持、按钮禁用
- 请求在进入真实执行链路前即被拒绝，未实际开始执行
- 仅用于页面展示的静态对象或占位数据

## 6. 数据落地方式与历史补录策略

### 6.1 落地方式

新增 Task Center 自己的统一任务表，用作统一列表和轻量详情的数据源。

这张表只存任务中心视角必需的信息，不复制所有原始业务细节。原模块仍保留完整业务数据。

优点：

- 分页、排序、筛选最稳定
- 三类任务查询口径一致
- 失败摘要和轻量详情可以稳定保存
- 后续扩展新的任务类型时，不需要重写整个查询层

### 6.2 Deploy 的接入方式

- 保持 Deploy 原任务数据为真实源
- 做一次历史补录，把现有 Deploy 任务映射到统一任务表
- 新 Deploy 任务创建和状态变化时，同步写入统一任务表

### 6.3 数据库连通性检测的接入方式

- 从 v0.0.6 起记录新增任务
- 单条检测生成一条统一任务
- 批量检测已选 / 当前页 / 全部筛选结果，各生成一条批任务
- 批任务摘要里记录成功、失败、跳过数量
- 单个数据库结果不单独进入统一任务列表，而是放入详情预览或原模块详情

### 6.4 Traffic 动作的接入方式

- 从 v0.0.6 起记录新增任务
- `preview`、`apply`、`rollback` 每次真实发起都生成一条统一任务
- 外部调用失败、业务失败、缺少 `rollbackToken`、状态回写失败都记为失败任务
- 摘要中保留动作类型、目标应用和结果摘要

### 6.5 历史补录边界

- Deploy：补历史
- 数据库连通性检测：不补历史，只记新增
- Traffic 动作：不补历史，只记新增

## 7. 前端交互设计

### 7.1 统一列表

Task Center 列表展示：

- 任务类型
- 任务名称
- 状态
- 发起人
- 开始时间
- 结束时间
- 摘要
- 操作

摘要示例：

- Deploy：`发布 app-a 到 prod，3 台主机，当前执行到 host-2`
- 数据库检测：`批量检测 20 条，成功 16，失败 3，跳过 1`
- Traffic：`Apply checkout-gateway，状态已写回，rollbackToken 可用`

### 7.2 筛选能力

v0.0.6 只做最小够用的筛选：

- 按任务类型
- 按状态
- 按时间范围
- 按关键词

关键词只匹配用户可见主字段：

- 任务名称
- 摘要
- 应用名 / 数据库名 / Deploy 名称等主标识

### 7.3 轻量详情抽屉

点击任务后，在 Task Center 内打开详情抽屉。

抽屉包含三块内容：

1. **基础信息**
   - 类型、状态、发起人、开始时间、结束时间

2. **任务摘要**
   - 用户可读的结果说明
   - 若失败，直接展示失败原因

3. **任务特定预览**
   - Deploy：应用、环境、主机数量、当前阶段或原始状态预览
   - 数据库检测：单条/批量、目标范围、成功/失败/跳过汇总
   - Traffic：动作类型、目标应用、策略、插件、`rollbackToken` 是否可用

抽屉底部提供一个深链按钮：

- `查看原始详情`

### 7.4 交互边界

明确不做：

- 抽屉内重试
- 抽屉内取消
- 抽屉内直接二次触发动作
- 多任务联动操作

## 8. 后端写入时机、状态流转和失败语义

### 8.1 写入时机

统一原则：**任务真正进入真实执行链路时创建记录，结果明确后更新记录。**

- 发起成功并进入真实执行链路时，创建统一任务记录，初始记 `running`
- 执行完成后，回写 `success` 或 `failed`
- 若请求根本没进入真实执行链路，则不创建任务

### 8.2 Deploy

- 继续使用原始 Deploy 执行链路
- 统一任务中心只做状态映射和数据同步
- 统一列表使用统一状态，详情预览中保留原始状态文本

### 8.3 数据库检测

**单条检测**
- 发起后建任务，记 `running`
- 检测成功记 `success`
- 检测失败记 `failed`

**批量检测**
- 整批发起后创建一条批任务，记 `running`
- 整批结束后统一结算
- 全部成功才记 `success`
- 只要出现失败，则整批记 `failed`
- 详情预览中保留成功数、失败数、跳过数

### 8.4 Traffic

- 真实动作发起后创建任务，记 `running`
- 外部服务成功且 EnvOps 状态回写成功后记 `success`
- 外部 REST 不可用、业务失败、缺少 `rollbackToken`、状态回写失败等都记 `failed`
- 不支持而未发起的对象不生成任务

### 8.5 失败信息层级

统一任务记录里保留两层失败信息：

1. **统一失败摘要**
   - 用于列表直接看懂失败
   - 示例：`traffic rest service is unavailable`

2. **详细失败预览**
   - 用于抽屉里展示更具体原因、计数或关键字段

## 9. 测试策略与验收口径

### 9.1 后端测试

至少覆盖：

- 统一任务中心查询测试：三类任务都能查到，分页与筛选正确
- 写入时机测试：真实发起时创建记录，完成后状态正确回写
- 不生成任务测试：不可执行对象不会进入任务中心
- 历史 Deploy 补录测试：补录后可正常展示和筛选
- 数据库单检失败测试
- 数据库批检部分失败时整批记 `failed` 的测试
- Traffic 外部调用失败测试
- Traffic `apply` 缺少 `rollbackToken` 的失败记录测试
- 三类任务详情预览字段完整性测试

### 9.2 前端测试

至少覆盖：

- 列表渲染：三类任务都能显示
- 筛选行为：类型、状态、关键词、时间范围都生效
- 详情抽屉：不同任务类型展示正确预览
- 失败展示：失败原因在列表和抽屉可见
- 深链跳转：原始详情入口正确
- 口径一致性：Deploy 不再是唯一任务类型，数据库批量检测是一批一条，Traffic 记录只来源于真实发起动作

### 9.3 验收通过标准

1. Task Center 列表可同时展示 Deploy、数据库检测、Traffic 三类任务
2. Deploy 历史记录可见
3. 数据库检测和 Traffic 从新发起开始进入 Task Center
4. 成功与失败任务都能在列表中看到
5. 数据库批量检测按“一批一条任务”呈现
6. 点击任务可打开轻量详情抽屉
7. 抽屉可查看基础信息、摘要、失败原因或汇总信息
8. 可从抽屉深链跳回原模块
9. 文档、文案、release notes 与实际行为一致

### 9.4 不属于本次验收的内容

- 完整统一详情页
- 任务重试或取消
- 多任务编排
- 数据库检测和 Traffic 的历史补录
- 新执行引擎或统一调度器抽象

## 10. 风险与控制

### 10.1 主要风险

- 三类任务真实状态与统一状态映射不一致
- 批量数据库检测的失败语义容易引起误解
- Deploy 历史补录可能出现字段缺失或旧状态不完整
- Task Center 轻量详情过少会像弱化版列表，过多又会膨胀成完整详情页

### 10.2 控制方式

- 先把统一状态严格收敛为四档，并在详情预览保留原始状态文本
- 明确数据库批检“有失败即整批失败”，同时保留成功/失败/跳过汇总
- 历史补录只做 Deploy，避免把不完整的数据库检测和 Traffic 历史硬补成脏数据
- 轻量详情只展示理解任务所必需的信息，深层内容统一回原模块

## 11. 版本结论

v0.0.6 应把 Task Center 从 deploy-only 队列视图收敛成最小真实统一任务中心。

这不是完整任务平台，也不是统一执行控制台，而是一个明确可交付、可验证、可继续扩展的中间层：

- 统一看见三类真实任务
- 统一理解它们的状态与结果
- 统一入口查看轻量详情
- 需要深入时再回原模块

这条路线最符合当前仓库的节奏，也最有机会在一个版本内做实。
